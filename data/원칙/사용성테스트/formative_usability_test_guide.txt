형성적 사용성 테스트(Usability Test, UT) 가이드

========================================

[UT 방법론의 포괄적 지침]

사용성 테스트(Usability Test, UT)는 참가자가 실제 사용 맥락과 유사한 환경에서 일련의 태스크를 수행하는 과정을 관찰하여, 프로덕트의 사용성을 평가하는 기법

참가자의 발언, 행동, 비언어적 반응을 종합적으로 기록하고 분석함으로써 프로덕트의 문제점, 개선 기회, 사용성 목표 달성 여부를 검증할 수 있음

========================================

[UT의 평가 요소]

사용성 평가 요소에 대한 상세 내용은 공통 가이드를 참조하세요: ux_research_common_guide.txt

형성적 사용성 테스트에서는 주로 다음 요소들을 중점적으로 평가합니다:
- 유효성: 태스크 성공률과 오류 발생률
- 효율성: 태스크 수행 시간과 인지적 부담
- 만족도: 사용 과정에서의 주관적 만족도

========================================

[UT의 장점과 한계]

장점:
- 참가자의 대답에 의존하지 않고 직접 행동을 관찰한다
- 적은 수의 참가자만으로도 대부분의 사용성 문제를 찾아낼 수 있다
- 데이터 분석 과정이 짧은 편이라 리서치 인사이트를 프로덕트에 빠르게 반영할 수 있다
- 상대적으로 시간과 비용이 적게 든다

한계:
- 질적 리서치의 특성상 통계적으로 유의미하지 않고, 결과 역시 일반화할 수 없다
- 테스트가 아닌 실제 환경에서 사용자들이 어떻게 행동할지 예측하지 못한다
- 테스트로 드러난 사용성 문제가 출시 후 얼마나 많은 사용자에게 영향을 미칠지 알 수 없다

========================================

[UT 일정 가이드]

리서치 진행 과정에 대한 공통 가이드는 ux_research_common_guide.txt를 참조하세요.

형성적 사용성 테스트 특화 사항:
준비(1~2주)
- 프로토타입 확인하기
- 시나리오와 태스크 만들기
- 디스커션 가이드 작성하기

테스트하기(1~2일)
- 시나리오와 태스크 전달하고 진행(세션 15~90분)

분석하기(1~2주)
- 레코딩 리뷰, 사용성 문제 발견

========================================

[UT 진행 형식 가이드]

1. 대면 UT

특징:
리서처와 참가자가 대면으로 진행하며, 리서처는 참가자의 행동을 관찰하고 추가 질문도 가능. 사용자 행동의 구체적인 맥락과 의도를 이해하고자 한다면 리서처가 진행하는 대면 인터뷰가 적합

장점:
실시간으로 이루어지기 때문에 참가자의 이해력과 배경 지식에 따라 인터뷰 시간을 조정하며, 추가적인 질문들을 자유롭게 수행 가능

단점:
시간 소요가 많아 리서치 할 수 있는 참가자 수가 제한적이다.

2. 원격 UT

특징:
리서처와 참가자가 원격으로 진행하며, Zoom 등과 같은 원격 미팅 툴 사용

장점:
시간 절약 및 참가자 모집에 지역적 제한이 없어, 다양한 배경의 사용자를 모집 가능

단점:
테스트 플랫폼의 기술적 문제로 진행 및 프로토타입이 제대로 작동하지 않을 위험

3. 리서처 없이 원격 UT

특징:
참가자가 미리 짜인 태스크를 스스로 진행하며 주어진 질문에 대답하는 형태.
리서처는 테스트 종료 후 레코딩 된 내용을 확인.
평가하고자 하는 프로토타입의 작업 흐름이 단순할 때 사용.
필요한 참가자 수가 많을 때 사용.

장점:
많은 수의 참가자 테스트 가능하며, 모더레이팅이 없으므로 진행 시간 절약 가능
리서처가 없기 때문에 참가자들이 더욱 솔직하게 생각과 의견을 표현할 수 있음

단점:
시나리오와 태스크가 이해하기 쉽도록 자세하게 설계되어야 함.
참가자가 이를 제대로 파악하지 못할 경우 데이터의 신빙성이 낮아짐.
추가 질문이 불가하고 참가자에게 전적으로 태스크를 맡기므로 피드백 퀄리티가 낮을 수 있음

========================================

[UT용 시나리오와 태스크 작성 가이드]

1. 현실적으로 작성한다
시나리오 및 태스크는 참가자가 일상에 겪을 법한 상황을 구체적으로 준비, 작성해야 함
태스크 간 흐름과 태스크에서 요청하는 내용 두 가지 모두 구체적으로 작성

예시: "고양이가 아플 때를 대비해서 반려동물 보험에 가입하기로 결정했습니다. 반려동물 의료보험 웹사이트에서 가장 적합한 보험상품을 찾아 결제까지 진행하도록 하겠습니다."

예시: 신용카드를 새로 만드는 시나리오 시 신용카드를 신청하기에 앞서 여러 신용카드 정보를 살피고 비교하는 과정이 포함된 흐름을 구성해야 함.

2. 구체적으로 작성한다
참여자에게 제공할 태스크의 목적을 자세하고 명확하게 밝혀야 함

예시:
- 나쁜 예시: "신용카드를 만들어 보세요"
- 좋은 예시: "이번에 자동차를 새로 사게 되었다고 가정하겠습니다. 운전하면서 다양한 혜택을 받을 수 있는 신용카드를 골라 보시겠어요?"

3. 힌트를 주지 않는다
구체적으로 태스크를 전달하되 '인터페이스'에 쓰인 단어를 사용하지 않아야 함

예시:
나쁜 예시: "주유 할인을 받을 수 있는 신용카드를 찾아보세요"와 같이 정확한 인터페이스 상의 단어를 사용할 시 참가자는 '주유 할인'이라는 키워드가 포함된 카드를 바로 고르게 될 수 있으니 사용하지 않는다.

========================================

[UT 목적과 주의사항 안내 가이드]

참가자가 아닌 프로덕트 평가라는 점을 강조하여 안내해야 함
사용성 테스트는 프로덕트 콘셉을 테스트하는 데 목적이 있기 때문에, 참가자가 태스크를 제대로 수행하지 못하거나 실행하며 헤매는 것은 참가자 능력과 상관없는 프로덕트 문제임을 알려야 함

솔직한 평가를 부탁해야 함
참여자는 상대의 기분을 상하게 하지 않기 위해 긍정적으로 평가하는 경향이 있음
따라서 가감없는 솔직한 피드백이 프로덕트 개선에 도움이 된다고 반드시 설명해야 함
리서처는 프로토타입을 디자인한 사람이 아니라서 객관적인 평가에도 상처받지 않는다고 강조하는 것도 좋은 방법

소리 내어 생각하기 (Think Aloud)를 요청해야 함
참가자에게 머릿속에 떠오르는 생각과 감정을 모두 공유해 달라고 요청
참가자는 태스크 수행에 집중하면 속으로만 생각할 수 있기 때문에 의식적으로 작은 감정이라도 스스럼없이 말해달라고 요청해야 함

프로토타입이 가지는 한계를 설명해야 함
프로토타입은 테스트 목적으로 제작되었기 때문에 평소 사용하는 일반 프로덕트와는 달리 일부분만 사용할 수 있다는 점을 설명

원하는 피드백이 있다면 도입부에 미리 언급해야 함
빠르게 태스크를 수행하기 보다 천천히 실행하며 의견을 충분히 전달해 달라고 요청하거나, 카피를 눈여겨 봐 달라고 하는 등 기대하는 피드백 내용을 확실히 전달해야 함

========================================

[UT 시나리오와 태스크 전달 및 진행 가이드]

참여자에게 질문 시 단어 하나도 신중히 선택해야 함
인터페이스에 포함되어 있는 단어를 사용하지 않고 질문해야 함
리서처가 테스트 도중 사용하는 단어는 참가자에게 무의식적으로 영향을 미칠 수 있음

- 나쁜 예시: "다시 뒤로 돌아가려면 어디를 누르시겠어요?"
- 좋은 예시: "이전에 보던 페이지로 가려면 어디를 누르시겠어요?"

참가자의 발언보다 행동에 주목해야 함
참가자가 프로덕트에 대한 호감/반감을 가질 수 있음
리서처가 바로 옆에 있어서 부정적인 피드백을 못하거나, 헤매는 모습을 보이기 싫어하는 경우도 발생할 수 있음
따라서 참여자의 말 보다는 행동을 관찰하는 것에 중점을 둬야함

참가자에게 무엇을 기대하는지 물어야 함
참가자가 프로덕트의 특정 링크나 버튼을 누르려고 하면 그다음에 어떤 화면과 내용이 나올 것 같은지 물어봄을 통해서 버튼의 이름이 역할을 제대로 하는지 평가할 수 있음

'왜'라는 질문을 적극 활용해야 함
참가자의 행동과 대답의 이유에 대해 추가 질문하는 것은 참가자의 태도와 생각을 알아내는 데 효과적임
참여자의 내적 생각과 니즈를 파악하는데 효과적인 방법임

참가자의 실수에 집중해야 함
참가자의 실수를 지적하지 않고, 참여자 스스로 인지할 때 까지 기다려야 함
참가자가 알아차리지 못한다면 충분한 시간을 기다린 후 어떤 생각으로 행동한 것인지 질문해야 함
이를 통해 사람들이 기대하는 바와 프로덕트의 제공 경험 사이의 간극을 발견할 수 있음

비언어적 커뮤니케이션에 주목해야 함
참가자의 목소리, 제스처, 표정 변화를 면밀히 파악해야 함

========================================

[UT 종료 후 데이터를 기반으로 사용성 문제 찾기 가이드]

사용성 테스트의 목적: 결과를 수량화하거나 미래를 예측하는 것이 아니라 최대한 많은 사용성 문제를 발견하고 그 가운데 무엇을 우선순위에 두고 개선할 것인지 결정하는 것

우선순위 결정 방법: 사용자 입장에서의 문제 심각도, 문제의 발생 빈도, 팀 개발 현황과 비즈니스 상황을 모두 고려하여 결정해야 함

이해관계자에게 전달 방법: 사용성 심각도 등급(Usability Severity Rating)을 활용하면 효과적임. 테스트에서 밝혀진 심각성 정도를 리포트에 표기해 프로덕트 팀이 참고할 수 있는 우선순위를 설정

========================================

[사용성 심각도 등급 예시]

중대한 문제 (Critical Issue): 심각한 사용성 문제로 엄청난 불편함을 야기하므로 프로덕트 출시 전 반드시 해결해야 함

온건한 문제 (Moderate Issue): 문제 정도가 심각하지는 않지만 태스크 수행 시간을 지연시키거나 불편을 일으킴

사소한 문제 (Minor Issue): 미미한 정도의 문제로 약간 불편할 수 있음

긍정적 피드백 (Positive Feedback): 참가자가 긍정적으로 반응했거나 의도한 대로 사용자 경험이 이루어졌음

참가자 제안 (Suggestion): 리서치 과정에서 참가자가 새롭게 제안한 내용

========================================

[효과적인 UT를 위한 핵심 원칙]

1. 사용자 중심 관찰
- 사용자의 행동과 반응에 집중
- 사용자의 말보다 행동을 우선시
- 비언어적 신호도 중요하게 다루기

2. 중립적 태도 유지
- 유도하지 않는 질문
- 사용자의 실수를 지적하지 않기
- 객관적 관찰자 역할 유지

3. 구체적 시나리오 활용
- 현실적인 상황 설정
- 명확한 목표 제시
- 힌트 없는 자연스러운 태스크

4. Think Aloud 활용
- 사용자의 내적 생각 공유 유도
- 의사결정 과정 파악
- 감정과 반응 수집

5. 체계적 분석
- 문제점 식별과 분류
- 심각도 평가
- 우선순위 설정
